# Knowledge System

## Overview

The knowledge system is Journey Buddi's memory of destination-specific information. It stores and retrieves the deep, local-expert-level knowledge that makes Buddi's guidance feel genuinely knowledgeable — the hidden viewpoints, the best time to visit specific spots, the stories behind the places, and the practical tips that only experienced travelers or locals know.

## Knowledge Categories

### 1. Attraction Knowledge
Core information about each attraction/activity:

```
AttractionKnowledge {
  id: string
  name: string
  destination: string
  location: { lat, lon, region, nearest_town }
  type: string[]              // ["hiking", "scenic", "wildlife"]
  description: string         // What it is and why it's special
  duration: { min, max }      // Time to spend
  difficulty: string          // easy, moderate, hard, extreme
  cost: { amount, currency, notes }
  availability: {
    seasonal: string          // "year-round", "oct-apr", etc.
    time_of_day: string       // "anytime", "morning", "evening"
    booking_required: boolean
    capacity_limited: boolean
  }
  conditions: {
    weather_sensitivity: string
    ideal_conditions: ConditionSet
    acceptable_conditions: ConditionSet
    dealbreaker_conditions: ConditionSet
    special_notes: string[]
  }
  logistics: {
    parking: string
    access: string            // How to get there
    facilities: string[]      // Toilets, shelter, food nearby
    cell_coverage: string     // "good", "patchy", "none"
  }
}
```

### 2. Pro Tips
Expert-level advice for each attraction:

```
ProTip {
  attraction_id: string
  category: "timing" | "viewpoint" | "preparation" | "hidden_gem" | "photography" | "food" | "logistics" | "safety"
  tip: string
  conditions_dependent: boolean  // Does this tip only apply in certain conditions?
  condition_context: string      // e.g., "at high tide", "on clear mornings"
  source: string                 // Where did this tip come from?
  confidence: string             // How reliable is this tip?
}
```

**Examples:**
```
{
  attraction_id: "nz-cathedral-cove",
  category: "timing",
  tip: "Arrive before 8 AM to have the beach almost to yourself. Tour buses start arriving at 9:30 AM and by 10 AM it's packed.",
  conditions_dependent: false
}

{
  attraction_id: "nz-milford-sound",
  category: "hidden_gem",
  tip: "On rainy days, temporary waterfalls appear all over the fiord walls — there can be hundreds. Some photographers prefer Milford in the rain for this reason.",
  conditions_dependent: true,
  condition_context: "during or after rain"
}

{
  attraction_id: "nz-tongariro-crossing",
  category: "viewpoint",
  tip: "The best photo of the Emerald Lakes is from the ridge 50m BEFORE the official viewpoint. You get the three lakes with Red Crater behind them. Most people walk past it.",
  conditions_dependent: false
}
```

## Knowledge Sources

### Primary: AI-Generated with Validation
The bulk of the knowledge base is generated by LLMs (Gemini/GPT-4) with prompts designed to extract specific, accurate, guidebook-quality information. This content is then validated for accuracy.

**Process:**
1. Prompt the LLM with structured queries: "For [attraction], provide: optimal conditions, top 5 pro tips, 3 hidden gems, key logistics..."
2. Cross-reference key facts against multiple sources
3. Flag uncertain or potentially outdated information
4. Store with confidence levels

### Secondary: Web-Sourced
Supplement AI knowledge with structured extraction from:
- Official tourism board websites
- DOC (Department of Conservation) for NZ trail data
- Government weather service historical data
- Travel blogs (for pro tips and hidden gems)

### Tertiary: Community-Contributed (Future)
In later phases, allow travelers to contribute tips and corrections:
- Post-trip feedback on tip accuracy
- New hidden gems discovered
- Updated logistics information
- Photo-validated viewpoint tips

## Knowledge Retrieval (RAG)

### Vector Embedding

All knowledge content is embedded using a text embedding model (e.g., `text-embedding-3-small` from OpenAI or equivalent) and stored in a vector database.

**Embedding strategy:**
- Each knowledge item is embedded as a standalone chunk
- Chunks are sized for completeness (one tip, one story, one logistics block)
- Metadata tags enable filtered search (destination, attraction, category)

### Retrieval Flow

```
User context (activity + conditions + preferences)
                    ↓
    Construct retrieval query
                    ↓
    Vector search (top-K relevant chunks)
                    ↓
    Filter by relevance score threshold
                    ↓
    Inject into LLM prompt as context
                    ↓
    LLM generates response grounded in retrieved knowledge
```

### Query Construction

For a daily briefing about "Whale Watching in Kaikoura on a calm, clear day":

```
retrieval_queries = [
    "whale watching Kaikoura tips and advice",
    "Kaikoura whale watching preparation packing",
    "Kaikoura whale watching best time photography",
    "Kaikoura area food restaurants recommendations",
    "whale watching calm conditions what to expect"
]
```

Each query returns top-5 relevant chunks. Deduplicated and combined as context.

### Vector Database Choice

**Recommended: Chroma (for MVP) or Pinecone (for production)**

- Chroma: Open source, easy to set up, perfect for development and small scale
- Pinecone: Managed service, scales effortlessly, better for production
- Both support metadata filtering, which is critical for destination/category scoping

## Knowledge Gaps

### Handling Unknown Information

When Buddi doesn't have specific knowledge:

1. **Acknowledge honestly:** "I don't have specific tips for this spot — it's a bit off the beaten track."
2. **Provide general advice:** Use category-level knowledge (general hiking tips if specific trail tips are missing)
3. **Suggest research:** "You might want to check the DOC website for the latest trail conditions."
4. **Flag for enrichment:** Unknown queries are logged to identify knowledge gaps to fill

### Knowledge Expansion Strategy

1. **Launch with NZ pilot**: Deep, comprehensive knowledge for one destination
2. **Template the structure**: The knowledge schema works for any destination
3. **Scale with AI generation**: When adding a new destination, generate the knowledge base with LLMs + validation
4. **Refine with usage**: User interactions reveal gaps; post-trip feedback validates accuracy
